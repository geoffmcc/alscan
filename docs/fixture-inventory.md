# ALScan .als Fixture Inventory

## Real `.als` Files and GitHub

**Never commit or upload real Ableton-authored `.als` files to GitHub, including
blank Sets, validation Sets, private projects, client projects, or
acceptance-test outputs.**

## Synthetic Fixture Policy

All `.als` files tracked in this repository are **synthetic parser fixtures**.
They are gzip-compressed XML approximating Ableton Live Set structure for
ALScan's parser, health-check, and merge-analysis testing.

**Synthetic fixtures are NOT Ableton-authored Sets:**
- They have not been opened, saved, or validated in Ableton Live.
- They are NOT suitable for validating .als writer output.
- Parser acceptance does NOT imply Ableton Live compatibility.
- They may not open in Ableton Live at all.

## Synthetic Live-Set Generation Experiment — Closed

A 2026-07 experiment attempted to generate structurally complete synthetic
`.als` files capable of opening in Ableton Live 12.4.2 by incrementally
correcting undocumented XML invariants discovered through comparison with
real reference Sets.

The experiment progressed through several structural corrections but
ultimately caused a fatal program error in Ableton Live 12.4.2.

**Outcome: synthetic Live Set generation is formally terminated.**
- Generated fixtures remain useful for parser and analysis testing only.
- No synthetic fixture is claimed to be Ableton-compatible.
- No further Live acceptance testing of generated fixtures is planned.
- Future automation should use Ableton's own APIs (e.g., Extensions SDK)
  rather than direct `.als` XML synthesis.
- Direct full `.als` synthesis is deferred indefinitely.

Repository enforcement is provided by `scripts/check_als_policy.py`.

## Fixture Categories

### Synthetic Parser Fixtures
- **Definition**: Generated gzip/XML files for testing ALScan's parser, health checks,
  merge analysis, and reporting. NOT created or saved by Ableton Live.
- **Scope**: Fast automated tests; parser behavior; edge cases; malformed structures.
- **NOT suitable for**: Validating .als writing; proving Ableton compatibility;
  acceptance testing with real Live Sets.

### In-Memory Test Fixtures
- **Definition**: Python objects (Snapshot, MergePlan, etc.) built directly in test code.
  No .als file on disk.
- **Scope**: Merge logic, identity matching, state transitions, verification.

### Ableton-Authored Validation Fixtures
- **Definition**: Real .als files created, saved, and reopened in the supported
  Ableton Live version.
- **Location**: NOT committed to the repository. Kept in a documented local
  validation directory. See `validation/` convention below.
- **Scope**: Manual acceptance testing; future writer validation; compatibility checks.

## Generator Script

**`tests/fixtures/generate.py`** — generates all 3 committed .als fixtures.
Run with: `python -m tests.fixtures.generate`

The script explicitly documents that ALL generated fixtures are synthetic
and have NOT been validated in Ableton Live.

Version attributes use Live 12 schema values:
- `MajorVersion="5"` (Ableton Set XML schema version, NOT product version)
- `MinorVersion="12.0_11000"` (product version + build suffix)
- `SchemaChangeCount="1"`
- `Creator="Ableton Live 12.3.2"`

## Live 12 XML Schema vs Product Version

Ableton's .als XML format uses two different version concepts:

| XML Attribute | Meaning | Real Value |
|--------------|---------|------------|
| `MajorVersion` | XML schema version (5 = Live 10/11/12 era) | `"5"` |
| `MinorVersion` | Product version + build number | `"12.0_12402"` |
| `SchemaChangeCount` | Schema revision counter | `"2"` (12.4.2) |
| `Creator` | Human-readable Live version | `"Ableton Live 12.4.2"` |
| `Revision` | Build git revision hash | `<redacted>` |

**Do NOT use product version numbers as `MajorVersion`.** Using `MajorVersion="12"`
causes Ableton Live to reject the Set with a misleading message about requiring
a newer version, because the XML schema version 12 does not exist.

## Structural Incompleteness of Synthetic Fixtures

Real Ableton Live 12 Sets contain ~70 top-level elements under `<LiveSet>` including:
`NextPointeeId`, `OverwriteProtectionNumber`, `LomId`, `MainTrack`, `PreHearTrack`,
`SendsPre`, `Scenes`, `Transport`, `GlobalQuantisation`, `ScaleInformation`,
`GroovePool`, `ViewStates`, `NoteAlgorithms`, and many more.

The synthetic generator produces only 4 top-level elements: `<Tempo>`,
`<TimeSignature>`, `<Locators>`, and `<Tracks>`.

**Synthetic fixtures should not be expected to open in Ableton Live**
regardless of whether the root metadata is correct. The structural
incompleteness is by design — these fixtures exist for ALScan's parser
and health-check testing, not for Ableton Live compatibility.

Future `.als` writing would require preserving the complete XML structure
of a real Ableton-authored foundation Set rather than constructing XML
from minimal templates.
